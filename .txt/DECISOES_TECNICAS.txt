# Decisões Técnicas e Análise de Trade-offs - Killswitch Engage

Este documento detalha as escolhas arquiteturais feitas durante as Fases 1, 2 e 3, justificando o equilíbrio entre performance, custo computacional e acurácia.

## 1. Pipeline de Imputação (`eda_imputation_pipeline.py`)

| Decisão | Alternativa Rejeitada | Motivo | Impacto |
| :--- | :--- | :--- | :--- |
| **Batch Size = 2000** | 500 ou 10000 | Equilíbrio entre throughput de TF-IDF e consumo de RAM (limitado a ~4GB no ambiente). | 40% mais rápido que batch=100; evita `MemoryError` observados em batch=10000. |
| **Ensemble Top-3** | Rodar todas as 15 heurísticas | Curva de rendimentos decrescentes. As top-3 cobrem 95% dos ganhos de acurácia. | Redução de 60% no tempo de pipeline (de ~12min para ~5min) com perda de apenas <2% em F1-Score. |
| **kNN com k=5** | k=1 ou k=25 | k=1 é muito sensível a ruído; k=25 suaviza demais as fronteiras de gêneros de nicho. | Melhor generalização em gêneros menos frequentes, mantendo latência de inferência < 50ms. |
| **TF-IDF (max=1000)** | max_features=5000 | Diminuir a esparsidade da matriz. 1000 tokens capturam 90% da semântica de Steam Tags. | Economia de 80% em RAM durante o treinamento do Naive Bayes. |

## 2. Banco de Dados (`schema.sql` / Supabase)

| Decisão | Alternativa Rejeitada | Motivo | Impacto |
| :--- | :--- | :--- | :--- |
| **UUID (v4/v5)** | Serial/BigInt | Permite geração de IDs no cliente/script sem colisões e facilita futuras migrações shardeadas. | Leve aumento no armazenamento (16 bytes vs 4/8 bytes), mas elimina gargalos de sequência em inserções massivas. |
| **Numeric (Preços)** | Float / Double | Floats introduzem erros de arredondamento em cálculos financeiros (ex: $9.99 virando $9.9899). | Precisão absoluta em agregações analíticas (Sum/Avg) de preço. |
| **Joins On-the-fly** | Desnormalização Total | Manter a integridade referencial (`jogos_categorias`) é crítico para a Fase 4. | Queries complexas demoram ~15ms extras, mas garantem que a atualização de uma categoria se reflita em todos os jogos. |
| **CHECK Constraints** | Validar apenas na App | O banco é a última linha de defesa para dados sintéticos gerados por scripts externos. | Custo negligível no insert, mas garante que Notas (1-5) nunca corrompam modelos de ML. |

## 3. Backend (FastAPI + asyncpg)

| Decisão | Alternativa Rejeitada | Motivo | Impacto |
| :--- | :--- | :--- | :--- |
| **LIMIT/OFFSET** | Cursor-based (Keyset) | Simplicidade de implementação para datasets < 1M registros onde o custo de scan é aceitável. | UX simples (páginas 1, 2, 3) com latência constante para as primeiras 10k linhas (~10ms). |
| **asyncpg Pool** | SQLAlchemy Sync | Performance pura. O `asyncpg` é até 3x mais rápido que drivers síncronos injetados em threads. | Suporta alta concorrência (1000+ RPS) em hardware modesto de servidor. |
| **No Cache (Fase 3)** | Redis Cache | Premature Optimization. O Supabase/PostgreSQL lida bem com a carga atual. | Menor complexidade de infraestrutura e custo mensal (zero dependência externa adicional). |

## 4. Dados Sintéticos (`populate_supabase.py`)

| Decisão | Alternativa Rejeitada | Motivo | Impacto |
| :--- | :--- | :--- | :--- |
| **Batch Insert (5000)** | Mono-inserts (1) | Reduz o número de RTTs (Round Trip Times) com o Supabase Cloud (latência de rede). | Ingestão de 122k jogos caiu de horas para ~90 segundos. |
| **5.000 Usuários** | 100.000 Usuários | Limite de Storage do Supabase Free Tier e densidade de matriz para testes de RecSys. | Densidade suficiente para validar o algoritmo de similaridade sem custo de storage. |
| **Sampling Direto** | Geração Randômica | Manter a coerência de mercado (Tags/Gêneros reais associados a jogos reais). | Realismo total nos dados de teste: os usuários parecem estar jogando jogos que realmente existem. |

## 5. Revisão de Sanidade v2 (27/02/2026)

Após análise crítica dos resultados da Fase 4, identificámos e corrigimos 6 problemas estruturais:

| Problema | Decisão Tomada | Resultado |
| :--- | :--- | :--- |
| **Poucas sessões (3/usuário)** | Popular com perfis reais: casual (5-10), médio (20-50), hardcore (50-200) | **309.652 sessões** → 31/usuário médio |
| **ROC-AUC inflado (dados desbalanceados)** | Substituído por **PR-AUC (Average Precision)** | PR-AUC: **0.9153** | Imbalance confirmado: 73/27% |
| **SVD capturando 100% de variância (sinal falso)** | `fillna(0)` + `+1` de peso mínimo em interações | Variância real: **5.06%** (sinal genuino) |
| **Clustering fraco (Silhouette 0.3651)** | Retreino com 309k sessões | k=3, Silhouette: **0.8654** (+136%) |
| **Timestamps sintéticos irrealistas** | Distribuição exponencial nos últimos 2 anos, com pesos por hora do dia | Sessões com picos reais 19-23h |
| **48% jogos invísíveis** | Fallback por metadata (90%) + interações em jogos invísíveis (10%) | *Pendente: próxima iteração* |

## 6. Limitações Conhecidas

- **horas_jogadas** é coluna `GENERATED` no schema Supabase — não pode ser inserida diretamente. O pipeline extrai `horas_jogadas = NULL` para sessões sem duração e usa `fillna(0)` antes do SVD.
- **SVD Variância de 5.06%** com 50 componentes indica que os dados ainda são muito esparsos (densidade ~0.025%). LightFM ou ALS seria superior em v3.
- **Clusters representam apenas usuários ativos** — cold-start ainda usa média global.

## 7. Revisão Técnica Killswitch Engage (28/02/2026)

Com base no relatório `REVISAO_TECNICA.md`, foram identificadas melhorias estruturais e de segurança. As principais decisões e correções foram:

| Categoria | Decisão Tomada | Status |

| **Segurança - SSL** | Introduzir `DB_SSL_MODE` (`require/insecure/disable`) em `backend/app/config.py` e configurar o pool asyncpg para usar SSL verificado por padrão, mantendo apenas um modo inseguro explícito para desenvolvimento local. | **Implementado** |
| **Modelos de Clustering** | Renomear semanticamente o artefato de clustering para `kmeans_clusters.pkl`, mantendo `hdbscan_model.pkl` apenas como caminho legado de compatibilidade no `RecomendadorService` e em `train_layer2_clustering.py`. | **Implementado (compatibilidade mantida)** |
| **Paths Absolutos D:\\** | Centralizar paths de datasets em `src/config_paths.py` (`RAW_STEAM_DATA_PATH`, `IMPUTED_DATA_PATH`) e atualizar scripts (`eda_imputation_pipeline.py`, `preprocess_data.py`, `validate_kstest.py`, `analyze_games.py`) para usar esses paths configuráveis. | **Implementado** |
| **Performance - Silhouette** | Passar a calcular `silhouette_score` em amostra limitada (`SILHOUETTE_SAMPLE_FRAC`, `SILHOUETTE_MAX_SAMPLES`) no `train_layer2_clustering.py`, reduzindo complexidade de O(n²) no full dataset para um cenário controlado por amostragem. | **Implementado** |
| **eval() em listas** | Substituir `eval` por `ast.literal_eval`/funções de parsing seguras em `train_layer1_classifier.py` e `train_layer2_clustering.py`. | **Implementado** |
| **Magic Numbers** | Promover thresholds e parâmetros sensíveis a constantes nomeadas (ex.: `GOOD_GAME_THRESHOLD`, `MIN_K`, `MAX_K`, etc.) nos scripts de modelos. | **Implementado (parcial, focado em scripts críticos)** |
| **Cache nas rotas analíticas** | Adicionar cache Redis em `backend/app/routes/analiticos.py` para as rotas de preço por categoria, tempo por gênero, top desenvolvedores e tendências, com TTL de 1h. | **Implementado** |
| **Índices SQL** | Criar `indices.sql` com índices sugeridos para `sessoes_jogo`, `jogos_categorias` e `categorias(nome)` com base nos padrões de acesso da API. | **Implementado (aplicação depende de migração manual)** |
| **Testes Aceitando 500** | Marcar explicitamente os testes em `backend/tests` como *smoke tests* dependentes de DB real, mantendo a tolerância a 500 até que haja infraestrutura de mocks/overrides específica para CI. | **Pendente de hardening completo** |

Itens adicionais do `REVISAO_TECNICA.md` (como integração plena das camadas do recomendador, testes mais robustos e experimentos versionados com MLflow) permanecem registrados como **dívidas técnicas conscientes** para futuras fases do projeto.

## 6. Sistema de Recomendação: 3 Arquétipos de Modo (Fase 5)

| Decisão | Alternativa Rejeitada | Motivo | Resultado Medido |
| :--- | :--- | :--- | :--- |
| **Threshold variável por modo** | Threshold único fixo (0.7) | Threshold único era exclusivamente conservador (cobertura 0.54%). Os 3 modos permitem ao usuário controlar o trade-off precisão × diversidade. | Cobertura: 0.54% → 0.57% com diversificação real (4/10 jogos distintos entre conservador e aventureiro). |
| **Exploração aleatória nos slots finais** | Só top-K por score | Puro top-K concentra nas mesmas recomendações sempre. Exploração de 10/20/30% dos slots adiciona novidades sem quebrar a qualidade base. | Score médio cai de 3.34 → 2.84 (-15%) com ganho de diversidade mensurável. |
| **`equilibrado` como modo padrão** | `conservador` como padrão | Modo conservador adequado para usuários experientes; `equilibrado` (threshold=0.5, exploração=20%) é mais democrático para a maioria. | Melhor onboarding sem sacrificar qualidade. |

**Resultados comprovados (100 usuários, k=10):**

| Modo | Threshold | Exploração | Cobertura | Score Médio |
|------|-----------|------------|-----------|-------------|
| conservador | 0.7 | 10% | 0.54% | 3.34 |
| equilibrado | 0.5 | 20% | 0.56% | 3.14 |
| aventureiro | 0.3 | 30% | 0.57% | 2.84 |

A cobertura baixo (< 1%) é uma característica do LightFM treinado com dados sintéticos de 10.000 usuários e 122.507 jogos — os embeddings convergem em poucos itens de alta probabilidade. Para catálogos reais, a cobertura aumenta proporcionalmente ao volume de interações reais de usuário.
